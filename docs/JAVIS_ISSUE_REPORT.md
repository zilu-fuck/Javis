# Javis 当前主要问题报告

> 评估日期：2026-06-06
> 评估范围：完整产品体验走查（UI/UX、稳定性、数据持久化、Agent 编排）

---

## 总体结论

Javis 暴露出的核心问题不是单纯的 UI 细节，而是：

> **Agent 内部执行状态、用户对话流程、任务追问机制、历史持久化和前端稳定性之间还没有形成一个连续、可靠、可理解的产品闭环。**

用户体验上会感觉：

- 一会儿像聊天软件
- 一会儿像任务调试器
- 一会儿像临时表单
- 一会儿又像后台日志面板
- 任务过程、补充回答、历史记录都不够可追溯

---

## 根本原因（前置）

在列出具体问题之前，先明确 5 个根因，后续每个问题都会标注对应的根因编号：

1. **Agent 编排层和 UX 层没有隔离** — 内部 JSON、capability、event bus、ReAct loop 泄露到 UI
2. **任务状态机没有统一** — 多个状态表述混乱，"等待授权"和"需要回答"语义不互斥，进度始终 0%（#8 和 #10 同根）
3. **对话流和任务流没有合并** — 用户聊天、Agent 追问、补充回答、执行日志分散在不同区域
4. **持久化恢复链路有 bug** — SQLite 基础设施已实现（9 套 migration），但启动时的数据恢复链路可能未正确触发，导致重启后数据"消失"
5. **前端渲染阻塞主线程** — 大段内容渲染、密集状态更新、日志无虚拟滚动导致 UI 卡顿

---

## 一、P0 级问题：必须优先修

### 1. 运行过程中频繁"未响应" [根因 #5]

软件执行任务时会频繁卡住或无响应。

可能原因：

- 日志列表没有虚拟滚动（已有 `react-window` 依赖但未使用）
- React 状态更新太密集（eventBus 事件未节流）
- 大段 Markdown 渲染阻塞（`marked.parse()` 同步执行）
- 日志刷新过于频繁

### 2. 重启后历史数据不显示 [根因 #4]

重新打开软件后，左侧的工作空间、项目、历史对话消失。

> ⚠️ **诊断修正**：项目已有完整的 SQLite 持久化基础设施（`database.rs`、9 套 migration、`task-history.ts`、`workspace-session.ts`、`recent-workspaces.ts`）。QA 证据（`docs/qa/2026-05-27/`）也证实了任务历史跨重启可恢复。**问题不是"没持久化"，而是启动时的数据恢复链路存在 bug 导致数据未正确加载到 UI。**

需排查：

- `App.tsx` 启动时是否正确调用了 `workspace-session.ts` 的 load 函数
- SQLite 写入是否在应用退出前完成（WAL checkpoint 时机）
- 侧边栏 mount 时是否从持久化存储恢复选中状态
- `TaskSnapshot` 是否包含了对话消息（可能存了任务元数据但没存消息体）

### 3. 补充回答流程割裂主对话且提交后不可见 [根因 #3]

> ⚠️ **合并原 #3 和 #4**：这是同一个问题的两个表现——追问流程脱离对话时间线导致内容断裂 + 不可追溯。

当前表现：

1. 追问出现后，看不到上面的原始聊天内容（用户失去上下文）
2. 用户感觉像是进入了一个临时会话
3. 补充回答提交后，原聊天内容回来了但刚才的回答不见了
4. 用户无法回看自己回答了什么问题、基于什么上下文

**正确体验**：所有影响 Agent 理解和执行的用户输入，必须作为正式消息保留在主对话时间线中。

```
用户：我想做一个可以识别 Wallpaper Engine 视频的播放器。
Javis：我需要确认几个细节：1.视频来源？2.是否需要分类封面？3.是否只做本地预览？
用户：自动扫描 Wallpaper Engine，本地预览，有名称、有分类。
Javis：收到，我会基于这些补充继续研究方案。
```

### 4. 任务执行失败时无错误恢复 [根因 #1 + #2] 🆕

API 调用失败或步骤执行出错时，用户看到的是一个空白或卡住的界面，没有重试、降级或错误说明。

需增加：
- 失败状态的明确 UI 展示
- 重试按钮
- 优雅降级（如 LLM 超时后显示部分结果）
- 错误信息用户可读化

### 5. capability 路由失败 [根因 #1]

日志里出现 `capability: unknown`、`ReAct loop`。规划器生成的步骤没有正确匹配到实际工具或能力。

可能原因：

- planner 生成了不存在的 capability
- capability 名称在 planner prompt 和 registry 中不一致
- router 没有兜底降级
- 普通分析步骤和工具调用步骤没有区分清楚

## 二、P1 级问题：严重影响体验

### 6. Agent 内部信息直接暴露给用户 [根因 #1]

> ⚠️ **从 P0 降级**：这是 UX 清晰度问题，不是崩溃/数据丢失问题。`ThreadView.tsx` 中 `commanderExpanded` 开关通过 `AgentDetailSections` 暴露了原始 Agent 细节，这本身是有意的调试功能，但默认不应展示给普通用户。

内部规划 JSON（`reasoning`、`steps`、`assignedAgentKind`、`toolName`、`capability`、`outputContextKey`、`successCriteria`）直接展示给用户。应该默认渲染为人类可读的步骤列表。

### 7. 追问没有给用户选项 [根因 #3]

Agent 直接让用户自己思考并输入，没有提供合适选项。

> 📝 `ask-user.ts` 已有 `choices?: string[]` 参数，问题在于 Commander prompt 没有充分利用，且 UI 未渲染为可点击选项。

对于产品设计、方案创作、代码架构类任务，追问应提供：推荐选项、默认选项、示例回答、"不确定，帮我决定"的兜底选项。

### 8. 状态表达混乱 [根因 #2]

同一阶段里出现多个状态：等待授权、需要您回答、待处理、运行中、规划中。语义不互斥。（与 #10 共享根因 #2）

### 9. 输入入口冲突 [根因 #3]

页面中同时存在多个输入位置：中间任务卡片的提交回答、底部主聊天输入框、活动日志里的提交回答。用户不知道应该在哪里回答。

### 10. 任务进度始终显示 0% [根因 #2]

任务经历了创建→规划→追问→补充→执行，但进度仍然是 0%。（与 #8 共享根因 #2）

### 11. 运行中主内容区空白太多 [根因 #1]

用户提交补充后主区域几乎空白。应展示执行步骤卡片，每步有状态：等待中/执行中/已完成/失败/需要补充。

### 12. 日志内容不适合普通用户 [根因 #1]

活动日志中出现大量开发者信息（`Task event bus recorded task creation`、`Dispatching step via capability: unknown`、`ReAct loop`）。

> 📝 `task-event-bus.ts` 中已有 `taskEventToLogEntry` 映射，但产出的仍是技术性描述。需要将默认级别改为用户可读文本，原始技术信息折叠到开发者模式。

### 13. 日志时间不正确

可能原因：后端 UTC 前端没转本地时间、时区不一致、显示接收时间而非事件发生时间。

### 14. SQLite 初始化期间无加载指示 [根因 #4] 🆕

`App.tsx` 启动时执行 13 组数据库迁移，是沉重的异步操作。在 `areDurableApprovalRecordsReady` 变为 `true` 之前，用户看到的是部分/空白 UI，没有任何加载状态提示。

## 三、P2 级问题：体验和信息架构优化

### 15. 左侧侧边栏信息架构混乱

混合了项目、工作空间、聊天、自动任务、知识库、失败任务、历史对话。用户很难判断当前处于哪个层级。

### 16. 标题截断和历史项不可辨认

侧边栏里多个历史项显示成 `Commander DA...`，无法分辨任务。需从首条用户消息提取前 30 字作为标题。

### 17. 页面布局空白过大

主区域存在大量空白，尤其规划卡片只占左上角、追问区域横向过长、运行中页面几乎没有过程展示。

### 18. 附件静默丢弃 🆕

`ChatView.tsx` 将图片限制为最多 5 张，超出部分被 `.slice(0, 5)` 静默丢弃，用户得不到任何提示。

### 19. 无删除确认机制 🆕

`deleteHistoryEntry` 和 `deleteScheduledTask` 立即销毁数据，没有确认对话框。

### 20. 流式文本在 Agent 切换时可能闪烁 🆕

`ThreadView.tsx` 中 `useRenderedStreamingText` 维护了 `resetPending`、`active`、`isStreaming`、`rawText` 之间的复杂关系。Commander 与 Verifier 之间的文本交接可能导致视觉闪烁或内容重复。

---

## 四、根本原因总结（与开头一致）

1. **Agent 编排层和 UX 层没有隔离** — 内部 JSON、capability、event bus、ReAct loop 泄露到 UI（影响 #5、#11、#12）
2. **任务状态机没有统一** — 多个状态表述混乱，进度始终 0%（影响 #8、#10）
3. **对话流和任务流没有合并** — 追问脱离主对话时间线，用户补充不可追溯（影响 #3、#7、#9）
4. **持久化恢复链路有 bug** — SQLite 基础设施已就绪，但启动恢复链路未正确触发（影响 #2、#14）
5. **前端渲染阻塞主线程** — 大段内容同步渲染、密集状态更新、无虚拟滚动（影响 #1、#20）
