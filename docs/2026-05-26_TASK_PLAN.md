## 架构约束：core 与 desktop 的边界

本次多项任务涉及 LLM 调用。关键约束：

```text
packages/core/                        apps/desktop/
  ↓ 纯 TypeScript，无 Tauri 依赖         ↓ 有 Tauri invoke、ModelProvider
  ↓ 可放：类型、常量、prompt template    ↓ 可放：实际 LLM 调用、文件 I/O
```

**规则**: 凡是需要 `ModelProvider.complete()` 的逻辑，必须放在 `apps/desktop/src/`。core 层只放不依赖 Tauri 的纯数据/纯函数（prompt 模板、评分 schema、术语表、类型定义）。

## P0-1：新增 ChineseReviewer Agent + 审校流水线

### 问题

当前 Verifier agent 只验证证据完整性（pass/warn/fail），不做语言质量审校。DeepSeek 生成的中文可能在内容正确的前提下，仍然存在模板化表达、机械句式、不自然的翻译腔。

### 方案

新增中文审校 Agent，位于生成输出和最终展示之间：

```text
LLM 生成 → ChineseReviewer 审校 → 最终输出
```

Agent 定义（`packages/core/src/agents.ts`）：

```typescript
{
  id: "agent-chinese-reviewer",
  kind: "chinese-reviewer",  // AgentKind 新增类型，见下文
  displayName: "中文审校",
  description: "输出中文自然度审校",
  allowedToolNames: [],  // 审校不经过工具路由，直接调用 ModelProvider.complete()
  systemPrompt: {
    en: "You are a Chinese style reviewer...",
    zhCN: `你是 Javis 的中文审校模块。请检查下面文本，只做轻度修改：
1. 去掉模板化表达（如"通过……能够……"、"首先其次最后"堆叠）。
2. 减少机械句式，让句长有自然变化。
3. 保留原意，不新增事实。
4. 技术术语保持英文原文，但首次出现时给出中文解释。
5. 返回审校后的完整文本，不要解释修改了什么。`,
  },
}
```

`AgentKind` 新增 `"chinese-reviewer"` 类型（`packages/core/src/index.ts:130-139`）。ChineseReviewer 不走 `allowedToolNames` 路由——它是在 LLM 输出后、通过 `ModelProvider.complete()` 直接调用的后处理步骤。

审校触发时机：

- 写作/润色类任务 → 必须走审校
- 代码/技术类任务 → 可选（不做语言审校，只做术语一致性检查）
- 图像提示词类 → 保留约束字段，不做改写

### 审校挂载点设计

当前 flow（`packages/core/src/index.ts`）没有"输出后处理"钩子。5.26 采用**最小侵入方案**：在 `apps/desktop/src/app-runtime.ts` 中，给 `ModelProvider.complete()` 包一层 `reviewChineseOutput()` 包装函数。所有经过 `app-runtime.ts` 的 LLM 调用——`planWithModelProvider()`、`verifyWithModelProvider()`——在拿到 response 后、返回给 flow 前，统一走审校。

```typescript
// apps/desktop/src/app-runtime.ts
async function completeWithChineseReview(
  prompt: string,
  options: CompletionOptions | undefined,
  modelProvider: ModelProvider,
  reviewMode: "full" | "terms-only" | "none",
): Promise<CompletionResult> {
  const result = await modelProvider.complete(prompt, options);
  if (reviewMode === "none") return result;
  if (!options?.locale?.toLowerCase().startsWith("zh")) return result;
  return reviewChineseStyle(result.text, modelProvider, reviewMode);
}
```

现有调用点改动：

| 调用点 | reviewMode | 说明 |
|---|---|---|
| `planWithModelProvider` | `"terms-only"` | Commander 计划是结构化 JSON，只做术语一致性 |
| `verifyWithModelProvider` | `"terms-only"` | Verifier 输出是 pass/warn/fail，只做术语一致性 |
| opencode proposal 输出 | `"none"` | 代码 diff 不改写，由 Rust 侧 lib.rs 的 locale prompt 直接控制 |

后续写作/润色类 workflow（5.27+）新增时，直接使用 `"full"` 模式。5.26 先用 `"terms-only"` 跑通管线。

### 架构注意

审校 Agent 的 prompt template 放入 core 层（纯文本、无副作用），但**调用逻辑在 desktop 层**。原因是 `ModelProvider` 只在 `apps/desktop` 层可达，core 层拿不到。审校是 `ModelProvider.complete()` 的输出后处理，不经过路由。

```text
packages/core/          apps/desktop/
  agents.ts               app-runtime.ts
    ↓ 定义 Agent 定义        ↓ planWithModelProvider()
  chinese-reviewer.ts        → modelProvider.complete()
    ↓ 审校 prompt template   → reviewChineseStyle(result, locale)
    ↓ 评分 schema            → 返回审校后文本
```

### 涉及文件

- `packages/core/src/agents.ts`（新增 ChineseReviewer agent 定义）
- `packages/core/src/chinese-reviewer.ts`（新增，审校 prompt template + 评分 schema，纯文本无副作用）
- `apps/desktop/src/app-runtime.ts`（在 Commander / Verifier / opencode proposal 输出后调用审校）

### 成功标准

- `demoAgents` 包含 ChineseReviewer
- 写作类任务输出经过审校，去掉典型模板句
- 技术类任务术语不被错误翻译
- 不改变现有 Verifier 行为
- Vitest 新增 3+ 审校相关测试

## P0-2：中文输入预处理层

### 问题

用户输入直接进路由，没有做中文意图结构化。用户说“这个怎么弄”、“帮我写得像论文一点”、“不要那么AI”——这种口语化、隐式要求的输入，直接给 DeepSeek 容易跑偏。

### 方案

在路由之前加一层轻量预处理。用一个小的 JSON Output 调用（低成本 Flash 模型即可），将用户原始输入结构化为：

```json
{
  "language": "zh-CN",
  "intent": "论文润色 | 代码排错 | 图像修改 | 技术解释 | 文档撰写",
  "user_style": "正式自然 | 简洁口语 | 游戏美术提示词 | 技术文档",
  "must_keep": ["原意", "主要内容", "代码逻辑"],
  "must_avoid": ["擅自扩写", "英文术语过多", "AI腔", "模板化开头"],
  "missing_info": ["需要润色的原文在哪里"]
}
```

关键设计决策：

- 预处理不阻塞主流程。如果预处理超时（2s）或失败，原始输入直接进入路由。
- 预处理结果注入到 `SharedTaskContext`（core 层，无依赖问题），供后续 Agent 使用。
- 使用 `ModelProvider.complete()` 调用——这要求预处理逻辑在 `apps/desktop` 层，因为 `ModelProvider` 不在 core 层。

### 架构注意

预处理必须放在 `apps/desktop` 层。`ModelProvider` 依赖 Tauri invoke，core 层访问不到。

5.26 分两阶段落地——先建基础设施，再接通 flow：

```text
5.26（本次）：
  1. 新建 input-preprocessor.ts + 单元测试（独立可跑）
  2. 在 shared-context.ts 新增 CONTEXT_KEYS.PREPROCESSED_INPUT
  3. 在 app-runtime.ts 创建 SharedTaskContext 实例并包一层 start wrapper：
     → 预处理 → 写入 sharedContext → coreRuntime.start(userGoal)

5.27+（后续）：
  → flow（runCodeReviewTask 等）通过 SharedTaskContext 读取预处理结果
  → 写作/润色类 workflow 根据 intent/user_style 调整行为
```

关键点：5.26 不修改 core 层 `start()` 签名，也不要求现有 flow 立即消费预处理结果。`SharedTaskContext` 由 `app-runtime.ts` 创建并持有引用，通过模块作用域在 wrapper 和 flow 间共享（与 `ModelProvider` 的模式一致——desktop 层创建、注入 core 层工具实现）。

```text
apps/desktop/src/
  app-runtime.ts
    → const sharedContext = createSharedTaskContext()
    → const coreRuntime = createFileScanTaskRuntime({ ... })
    → 返回的 runtime.start 是包装后的：
      → input-preprocessor.preprocess(userGoal, modelProvider)  // 2s timeout
      → 成功: sharedContext.set(CONTEXT_KEYS.PREPROCESSED_INPUT, result)
      → 失败/超时: 静默跳过
      → coreRuntime.start(userGoal)
```

### 涉及文件

- `apps/desktop/src/input-preprocessor.ts`（新增，预处理 prompt + ModelProvider 调用 + timeout guard）
- `packages/core/src/shared-context.ts`（可能新增预处理结果 ContextKey）

### 成功标准

- 口语化输入被正确识别为具体意图
- 预处理失败时不影响正常流程
- `SharedTaskContext` 中可读取预处理结果（`CONTEXT_KEYS.PREPROCESSED_INPUT`），flow 消费留到 5.27+
- Vitest 新增 3+ 预处理场景测试

## P1-1：术语表注入

### 问题

DeepSeek 不了解项目特定术语，容易把 Token 翻译成“令牌”、把 Agent 翻译成“代理”或“智能体”，造成阅读混淆。需要一套项目级术语表，在每次 LLM 调用时注入 system prompt 前缀。

### 方案

新建术语表模块：

```typescript
// packages/core/src/terminology.ts
export const JAVIS_TERMINOLOGY: Record<string, string> = {
  "Agent": "Agent（不翻译为代理或智能体，保留英文）",
  "Token": "Token（不翻译为令牌，论文中可写为 Token）",
  "workbench": "工作台",
  "confirmed write": "已确认写入（指经用户批准的写操作）",
  "dry run": "干运行（指预览执行计划但不实际修改文件）",
  "patch": "补丁（不翻译为修补程序）",
  "hunk": "差异块（指 unified diff 中的一个变更片段）",
  "diff": "差异（不翻译为差别或区别）",
  "workspace": "工作区",
  "approval": "审批",
  "proposal": "提案",
  "verifier": "验证器",
  "Commander": "Commander（指挥官，保留英文）",
  "open source": "开源",
};
```

注入方式：

1. 每次 `ModelProvider.complete()` 调用时，在 system prompt 前缀追加术语表。
2. 术语表放在 prompt 固定前缀位置，可命中 DeepSeek 上下文缓存。
3. 按 locale 选择是否注入中文术语表（en 环境下不注入中文解释）。

### 涉及文件

- `packages/core/src/terminology.ts`（新增）
- `apps/desktop/src/model-provider.ts`（complete 调用前注入术语表）
- `apps/desktop/src/app-runtime.ts`（`planWithModelProvider` / `verifyWithModelProvider` 注入术语表——这两个函数直接拼 prompt，不经过 model-provider 的 complete 包装）
- `apps/desktop/src-tauri/src/lib.rs`（opencode proposal prompt 注入术语表）

### 成功标准

- 术语表在所有 LLM 调用路径中生效
- 中英文术语映射不互相污染
- 术语表内容可命中 DeepSeek 上下文缓存（固定前缀位置）

## P1-2：中文质量内评

### 问题

没有量化方式判断“中文好不好”。需要在审校 Agent 内部加一个评分维度，只用于决定是否触发二次改写，不展示给用户。

### 方案

审校 Agent 在审校完成后输出内部评分（不展示给用户）：

```json
{
  "accuracy": 8,
  "naturalness": 7,
  "style_match": 9,
  "term_consistency": 8,
  "constraint_following": 9,
  "redundancy": 3,
  "needs_revision": false
}
```

评分标准：

| 维度 | 满分 | 说明 |
|---|---|---|
| accuracy | 10 | 是否答到点上 |
| naturalness | 10 | 读起来像不像中文母语者写的 |
| style_match | 10 | 是否符合用户要求的正式/口语/论文/提示词风格 |
| term_consistency | 10 | 专业术语有没有乱翻译 |
| constraint_following | 10 | 有没有擅自加内容、改格式 |
| redundancy | 10 | 废话和模板句的严重程度（10=完全没有） |

阈值：

- `needs_revision` 由 LLM 综合 6 个维度自主判定（非硬编码分数阈值）。6 个评分维度是给 LLM 的"思考指引"——强制 LLM 逐维度审视后再输出判定，比直接问"好不好"更可靠。
- `needs_revision = true` → 自动触发一次改写（最多 1 次，防止循环）
- score 仅用于内部控制，不暴露给用户

### 涉及文件

- `packages/core/src/chinese-reviewer.ts`（在 P0-1 创建的审校 prompt template 基础上，追加评分 schema + `needs_revision` 判定 prompt）
- `apps/desktop/src/app-runtime.ts`（审校完成后解析评分、判断是否二次改写）
- `packages/core/src/chinese-reviewer.test.ts`（新增，评分 schema 验证 + prompt 模板测试）

### 成功标准

- 审校 Agent 输出内部评分
- 低分触发自动改写，改写后不再循环
- 评分不泄漏到 UI
