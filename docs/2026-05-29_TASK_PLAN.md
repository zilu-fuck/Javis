## P0-1：流式输出 UI 消费端（~4-5h）

### 当前状态

Rust SSE 后端已完成（`streaming.rs` 247 行，后台线程 → Tauri events → `stream-model-chunk` / `stream-model-done` / `stream-model-error`）。前端有：
- `useStreamingSnapshot` hook（`packages/ui/src/hooks/useStreamingSnapshot.ts`）
- `useSmoothStream` hook（`packages/ui/src/hooks/useSmoothStream.ts`）
- `StreamingMessage` 组件（`packages/ui/src/components/StreamingMessage.tsx`）

但 **task-flow 中 Agent 的实际输出仍走 `complete_model_prompt`（非流式）**，`stream_model_prompt_start` 未被 task-flow 消费。

### 方案

```
Agent 执行 tool call
    → app-runtime.ts 检测 agent.supportsStreaming
    → 走 stream_model_prompt_start（而非 complete_model_prompt）
    → 前端 listen("stream-model-chunk") → delta-reducer 累积
    → ChatView 中 Agent 消息实时增量渲染
```

### 步骤

| 步骤 | 文件 | 说明 |
|------|------|------|
| 流式调用切换 | `app-runtime.ts` | `executeAgentStep()` 中按 agent capability 选择 stream vs complete |
| delta → snapshot 管线 | `delta-reducer.ts` | 确认现有 delta-reducer 能处理流式增量 |
| ChatView 流式消费 | `ChatView.tsx` | Agent 消息气泡实时更新，不等待 complete |
| 取消 + 重试 | `ChatView.tsx` | 停止生成按钮 + 错误重试 |
| 降级策略 | `app-runtime.ts` | stream 失败时自动 fallback 到 complete |

### 成功标准

- Agent 输出逐字/逐 chunk 出现在 ChatView 中
- 用户可中途取消生成
- 流式失败时静默降级到非流式
- `pnpm check` 通过

> **范围控制**：优先做 ChatView 中 Agent 消息的流式增量渲染（步骤 1-3）。取消/重试（步骤 4）和降级策略（步骤 5）如时间不够移入 P2。最坏情况下，对话流式可见即为 MVP 交付。
