# Javis Desktop UI Detector Dataset Enhancement Plan

## Goal

Improve `artifacts/local-vision/yolo26n-ui.onnx` from a mobile/web GUI baseline into a stronger desktop UI candidate detector for Javis Computer Use.

Current status: Phase 3 is complete. `showlab/ShowUI-desktop`, a representative `ServiceNow/GroundCUA` subset, and a weakly labeled Javis-native screenshot bootstrap set have been converted/trained. The exported model has replaced `artifacts/local-vision/yolo26n-ui.onnx`.

Success means:

1. Keep the current YOLO26 UI model usable for smoke and benchmark.
2. Add desktop screenshot data with bounding boxes.
3. Convert added data to standard YOLO detection format.
4. Fine-tune from the current UI checkpoint, then export `yolo26n-ui.onnx` with `end2end=False`.
5. Verify with Javis screenshots through `local-vision:doctor`, `local-vision:smoke`, and `local-vision:benchmark`.

## Assumptions

1. The current baseline comes from `Rodrigomgs24/Dissertacao_GUI_Detection` and is trained on Rico + WebUI GUI data.
2. The first desktop add-on dataset should be small enough to download and convert locally.
3. Full training on this machine uses CUDA device 0, `NVIDIA GeForce RTX 5060 Ti`, with `torch 2.11.0+cu128` and `ultralytics 8.4.61`.
4. YOLO remains a candidate detector only. It should not directly authorize clicks or sensitive actions.

## Dataset Order

### Phase 1: ShowUI Desktop

Use `showlab/ShowUI-desktop` first.

Why:

1. Desktop-oriented GUI grounding data.
2. Moderate size compared with GroundCUA.
3. Contains normalized bounding boxes that can be converted to YOLO.

Limitations:

1. It is grounding data, not a native object detection dataset.
2. Element labels are not always one-to-one with Javis's 12 UI classes.
3. Class mapping must be conservative and auditable.

### Phase 2: GroundCUA

Use GroundCUA after the ShowUI conversion is proven.

Why:

1. More desktop-heavy.
2. Much larger: useful for stronger generalization across desktop apps.

Limitations:

1. Larger download and conversion cost.
2. Needs a dedicated converter and validation pass before training.
3. The local run used a stable representative subset because Hugging Face per-file conversion was slow and the first long conversion was stopped.

## Execution Plan

1. Convert ShowUI desktop to YOLO:
   - Download/read the dataset.
   - Save images under `artifacts/local-vision/datasets/showui-desktop-yolo/images/{train,val}`.
   - Save labels under `artifacts/local-vision/datasets/showui-desktop-yolo/labels/{train,val}`.
   - Generate `data.yaml`.
   - Write `conversion-report.json`.

2. Run a dataset smoke:
   - Validate image/label counts.
   - Validate class IDs.
   - Validate bbox normalization.

3. Run a tiny fine-tune smoke:
   - Start from the existing GUI checkpoint.
   - Use a tiny subset and one epoch.
   - Verify that Ultralytics can load the converted data.

4. Export smoke checkpoint:
   - Export with `format=onnx imgsz=640 end2end=False`.
   - Run Javis doctor/smoke/benchmark.

5. Decide whether to run longer fine-tune:
   - Only run full training when GPU or enough time is available.
   - On CPU, prefer short experiments and keep commands reproducible.

6. Convert and train a GroundCUA subset:
   - Use `scripts/prepare-groundcua-yolo.py`.
   - Audit final image/label counts before training because long Hugging Face per-file conversions can be interrupted.
   - Fine-tune from the current UI checkpoint.

7. Build a Javis bootstrap set:
   - Use `scripts/prepare-javis-bootstrap-yolo.py`.
   - Treat generated labels as weak supervision.
   - Fine-tune for a short run from the GroundCUA checkpoint.

8. Export and publish:
   - Export the final checkpoint with `format=onnx imgsz=640 end2end=False`.
   - Copy final `best.onnx` to `artifacts/local-vision/yolo26n-ui.onnx`.
   - Copy final `best.pt` to `artifacts/local-vision/yolo26n-ui.pt`.

## Risks

1. Weak label mapping can hurt the detector. Keep conversion reports and skipped counts.
2. Desktop grounding data may contain target-specific boxes rather than exhaustive element boxes.
3. CPU training can be too slow for full runs.
4. Evaluation datasets should not be mixed into training.
