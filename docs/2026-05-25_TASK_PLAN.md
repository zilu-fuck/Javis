# 2026-05-25 Task Plan

Last updated: 2026-05-25

本文档记录 2026-05-25 的 Javis 改造任务方案。目标不是一次性重写完整多 Agent 架构，而是结合现有修复计划与 Proma 的可借鉴模式，先补齐 Javis 走向可靠多 Agent 运行时所需的地基。

## 总目标

把 Javis 从当前的硬编码单流 runtime，推进到更稳定、可扩展、可审计的基础形态：

- Runtime 生命周期稳定。
- 权限请求可按 requestId 精确恢复，不再依赖单例 handler。
- 调度逻辑有测试覆盖。
- 初步引入事件总线，降低 flow 与 UI snapshot 的耦合。
- `read-current-project` workflow 有最小可执行路径。

Proma 的经验可借鉴，但不照搬。Javis 当前更需要小步落地：先修状态、权限、事件和最小 workflow，再引入完整 Commander/Verifier LLM。

## Proma 分析摘要

深度分析了 [ErlichLiu/Proma](https://github.com/ErlichLiu/Proma)（基于 Claude Agent SDK 的 Electron 桌面 Agent 应用），提炼出 4 个可直接借鉴的模式：

### 最值得借鉴的 4 件事

| # | 模式 | Proma 实现 | Javis 落地方式 |
|---|------|-----------|---------------|
| 1 | **EventBus 解耦** | `AgentEventBus`（93 行）：中间件链 + handler 注册，`emit()` → middleware → handlers | Javis 加轻量 `TaskEventBus`，工具调用、权限请求、agent 状态、日志先变成事件，再由 snapshot reducer 汇总成 UI 状态 |
| 2 | **Permission Promise + Map** | `sessionId/requestId → { resolve, request }` Map，支持多会话并行等待 | 将 `pendingPermissionHandler` 单例改为 `Map<requestId, PendingPermissionHandler>`，精确解析避免互相覆盖 |
| 3 | **Session/JSONL 追加日志** | `~/.proma/agent-sessions/{uuid}.jsonl`，追加写入，支持 resume 和审计 | 先不全量迁移，只给 agent run / tool call audit 增加 JSONL 事件日志，用于恢复和审计 |
| 4 | **Provider Adapter** | `ProviderAdapter` 接口统一 `sendMessage()` 流式方法，注册表按 providerId 查找 | Javis 把 `ModelProvider.complete()` 真正接通，再让 Commander / Verifier / Code 共用同一接口 |

### 暂不照搬的点

- **多 Agent 依赖 SDK 内置 Agent 工具**：Proma 的子代理通过 Claude SDK `agents` 选项注入，主 Agent 通过 Agent 工具按名称调用。Javis 用 opencode/Tauri 自己编排，需要自己的 workflow executor，不能只复制 prompt。
- **子代理工具自动批准**：Proma 对 `options.agentID` 存在时直接放行所有工具。Javis 应保留 read 自动、preview/confirmed_write 显式审批，不因为是 sub-agent 就放行写操作。
- **大型 orchestrator**：Proma 的 `agent-orchestrator.ts` 1300 行。Javis 应避免把复杂度集中到一个超级 orchestrator，渐进式扩展。
- **完整会话 rewind / resume**：需要 SDK 级文件快照支持，Javis 当前不做。
- **Claude SDK preset 依赖**：Proma 的系统提示词基于 `preset: 'claude_code'`。Javis 需要自己构建提示词。

## P0-1：修 Runtime 生命周期 bug

### 问题

`apps/desktop/src/App.tsx:283` 的 runtime subscribe effect 依赖 `workspacePath`，但 cleanup 中会调用 `runtime.dispose()`。切换 workspace 时可能误 dispose 当前 runtime，导致订阅或后续 task 更新异常。

### 方案

- 从 subscribe effect 依赖里移除 `workspacePath`。
- 回调内使用 `workspaceRef.current` 读取最新 workspace。
- 保持 runtime 只随 `modelSettings` 重建。

### 涉及文件

- `apps/desktop/src/App.tsx`

### 验证

```bash
pnpm -r typecheck
pnpm -r --if-present test
```

### 成功标准

- 切换 workspace 不会导致 runtime 被误销毁。
- task snapshot 订阅仍正常更新。
- 现有测试全部通过。

## P1-3：将 `pendingPermissionHandler` 改为 Map

### 问题

`packages/core/src/index.ts:383` 当前只有一个 `pendingPermissionHandler`。未来 workflow executor 支持并行或多个 Agent 同时等待权限时，后一个权限请求会覆盖前一个 handler。

### 方案

改为 requestId 维度的权限 handler map：

```typescript
const permissionHandlers = new Map<string, PendingPermissionHandler>();
```

修改点：

- `TaskRuntime.resolvePermission(decision, requestId?)` 支持可选 requestId。
- 创建 permission request 时执行 `permissionHandlers.set(request.id, handler)`。
- resolve 后删除对应 handler。
- 保留无 requestId 的 legacy fallback，用于现有 UI 调用路径。

### 涉及文件

- `packages/core/src/index.ts`

### 验证

- 更新 PDF organization permission 流程测试。
- 更新 Code Agent preview / apply permission 流程测试。
- 新增一个并发权限 handler 的单元测试，证明两个 requestId 不互相覆盖。

### 成功标准

- 权限请求可按 requestId 精确 resolve。
- 旧 UI 不传 requestId 时仍兼容。
- 现有 PDF 和 Code Agent 审批流程不回归。

## P1-4：引入轻量 `TaskEventBus`

### 问题

当前 flow 直接手写 `TaskSnapshot`，agent 状态、日志和工具调用状态都散落在各个 flow 中。借鉴 Proma 的 `AgentEventBus`（93 行，中间件链 + handler 注册），先做最小可用版本。

### 方案

新增：

```
packages/core/src/task-event-bus.ts
```

先定义最小事件集合：

```typescript
type TaskEvent =
  | { kind: "task.created"; taskId: string }
  | { kind: "agent.status"; taskId: string; agentKind: AgentKind; status: AgentRunStatus; message: string }
  | { kind: "tool.planned"; taskId: string; toolName: string; detail: string }
  | { kind: "permission.requested"; taskId: string; request: ToolPermissionRequest }
  | { kind: "permission.resolved"; taskId: string; requestId: string; decision: "approved" | "denied" }
  | { kind: "task.completed"; taskId: string }
  | { kind: "task.failed"; taskId: string; error: string };
```

5.25 只接入一条最小路径，优先选择 Code Review 或 Project Inspection。不要一次性改完所有 flow。

### 涉及文件

- `packages/core/src/task-event-bus.ts`（新增）
- `packages/core/src/index.ts`（emit 事件）

### 成功标准

- 至少一个 flow 的日志或 agent 状态可由事件派生。
- 不改变 UI 对 `TaskSnapshot` 的消费方式。
- 不引入新的全局 orchestrator 大文件。
