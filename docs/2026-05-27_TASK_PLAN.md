## P0-1：Live Code Agent QA — reqwest 请求调试

### 问题

5.26 的 fixture QA（deny + approve）全部通过，但真实 DeepSeek 凭证下是 fail-closed：proposal 生成不产生可解析输出，不尝试文件写入（安全失败）。

Python 验证脚本（`test_fallback.py`）可以拿到 HTTP 200 + 有效 JSON proposal，Rust 代码不行。

### 诊断现状

项目使用的是 `reqwest 0.12` + `native-tls`（Windows Schannel），不是 ureq。问题可能在于：

1. **请求构造差异**：Rust 代码发送的 headers/body 与 Python 脚本不同（例如 DeepSeek 的 `thinking` 参数、`stream` 参数）
2. **响应格式不匹配**：DeepSeek API 可能返回 OpenAI 兼容的流式 SSE 格式，但 Rust parser 按非 streaming JSON 格式解析
3. **native-tls 证书问题**：Schannel 在某些 Windows 配置下缺少根证书

### Proma 项目的参考

`github.com/ErlichLiu/Proma` 是一个 TypeScript/Electron 项目，使用 DeepSeek 的 **Anthropic 兼容端点**（`https://api.deepseek.com/anthropic`）而非 OpenAI 兼容端点（`https://api.deepseek.com`）。关键设计：

- DeepSeek 注册为 `AnthropicAdapter('deepseek')`，走 `/v1/messages` 协议
- SSE 解析委托给 adapter（`parseSSELine`），统一处理 `content`、`reasoning_content`、`tool_calls`
- HTTP 层是原生 `fetch`（undici），无 reqwest 相关问题

Javis 走 OpenAI 兼容端点（`/chat/completions`），两者不冲突。但 Proma 的 adapter 注册模式值得后续引入多 provider 时参考。

### 方案

分三步排查：

**Step 1：加 diagnostics 日志**（跳过——直接定位到根因）

**Step 2：对比 Python 请求**（跳过——直接定位到根因）

**Step 3：根因确认 → 修复**

根因：`default_openai_compatible_base_url_for_provider("deepseek")` 返回 `"https://api.deepseek.com"`（缺少 `/v1`），导致 `create_chat_completions_endpoint` 拼接出 `"https://api.deepseek.com/chat/completions"` 而非正确的 `"https://api.deepseek.com/v1/chat/completions"`。

Python 脚本使用完整路径（或 openai 库自动补全 `/v1`），所以能拿到 HTTP 200；Rust 代码请求到错误 URL 拿不到有效响应。

修复（commit pending）：
```rust
// lib.rs:1698 — 两处 default base URL 函数
"deepseek" => "https://api.deepseek.com/v1".to_string(),  // 加了 /v1
```

新增测试断言：验证 `default_openai_compatible_base_url_for_provider("deepseek")` 返回含 `/v1` 的 URL，且 `create_chat_completions_endpoint` 拼接后为正确路径。

## P0-2：localStorage → SQLite 数据迁移

### 问题

SQLite 基础设施已完成（`database.rs`、Tauri 命令 `db_execute`/`db_select`、migration 系统 `schema_migrations` 表、model-settings 已迁入），但以下三个模块仍在使用 localStorage：

| 模块 | 文件 | 数据 | 目标表 |
|------|------|------|--------|
| 任务历史 | `apps/desktop/src/task-history.ts` | 已完成/失败/取消的任务快照 | `task_history` |
| 审批记录 | `apps/desktop/src/approval-records.ts` | PDF 和 Code Patch 的持久化审批 | `approval_records` |
| 最近工作区 | `apps/desktop/src/recent-workspaces.ts` | 最近使用的工作区路径列表 | `recent_workspaces` |

### 方案

照搬 `model-settings-persistence.ts` 的模式：

1. 在每个模块中新增 SQLite 读写函数（`loadFromDatabase` / `saveToDatabase`）
2. 首次启动时检测 localStorage 中有旧数据 → 迁移到 SQLite → 删除 localStorage key
3. 后续读写直接走 SQLite

migration 版本号递增，`model-settings` 是 v1，这三个对应 v2-v4。
